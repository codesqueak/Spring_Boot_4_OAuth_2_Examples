# OAuth2 Authorization Server — Integration Guide

This service acts as an OAuth2 / OpenID Connect authorization server.  
Base URL: `http://localhost:9095`  
Discovery document: `http://localhost:9095/.well-known/openid-configuration`

---

## Swagger UI

The interactive API documentation is available at:

```
http://localhost:9095/swagger-ui/index.html
```

### Accessing the Swagger page

1. Start the service, then open `http://localhost:9095/swagger-ui/index.html` in a browser.
2. You will see a list of available API endpoints grouped by controller.

### Authorizing requests in Swagger

Protected endpoints require a Bearer token. To authenticate:

1. Obtain a token using one of the grant flows below (e.g. the `/user/v1/login` endpoint visible in the UI itself).
2. Click the **Authorize** button (padlock icon) at the top right of the Swagger page.
3. In the **Value** field enter `Bearer <your_token>` and click **Authorize**.

All subsequent requests made from the Swagger UI will include the `Authorization: Bearer` header automatically.

### Required headers

The `x-correlation-id` header is required on all requests. Swagger does not add it automatically — use the **Parameters** section on each operation to supply a UUID value, e.g.:

```
x-correlation-id: $(uuidgen)   # or any UUID string
```

## Grant type 1 — Authorization Code (user-facing apps)

Use this when a human user must log in (web apps, SPAs, mobile).

### Step 1 — Generate PKCE parameters

This server requires **PKCE** (Proof Key for Code Exchange, [RFC 7636](https://datatracker.ietf.org/doc/html/rfc7636)).

PKCE protects against **authorization code interception attacks**: if an attacker somehow intercepts the `code` from the redirect URL (e.g. via a malicious app registered on the same redirect URI, browser history, or a referrer header leak), they cannot exchange it for tokens because they don't know the `code_verifier` that was only ever held in memory by the legitimate client.

The mechanism works as follows:
1. The client generates a random `code_verifier` and derives a `code_challenge` from it (SHA-256 hash, base64url-encoded).
2. The `code_challenge` is sent in the authorization request — the server stores it alongside the issued `code`.
3. When exchanging the `code` for tokens, the client sends the original `code_verifier`. The server hashes it and checks it matches the stored `code_challenge`. If it doesn't match, the request is rejected.

An intercepted `code` is therefore useless without the `code_verifier`, which never leaves the client.

Generate the verifier and challenge before redirecting:

```bash
CODE_VERIFIER=$(openssl rand -base64 32 | tr -d '+/=' | head -c 43)
CODE_CHALLENGE=$(echo -n "$CODE_VERIFIER" | openssl dgst -sha256 -binary | openssl base64 | tr '+/' '-_' | tr -d '=')
```

Save `CODE_VERIFIER` in the user's session — you will need it in Step 2.

### Step 2 — Redirect the user to the authorization endpoint

In a browser ... make sure you don't copy in any extras spaces !

```
http://localhost:9095/oauth2/authorize
?response_type=code
&client_id=pan-client
&redirect_uri=http://localhost:3000/callback
&scope=openid profile email pan.read
&state=random-csrf-token
&code_challenge_method=S256
&code_challenge=CODE_CHALLENGE
```


#### Parameter reference

| Parameter | Value | Purpose |
|---|---|---|
| `response_type` | `code` | Requests the Authorization Code grant. The server returns a short-lived `code` in the redirect; the client exchanges it for tokens via a back-channel POST. |
| `client_id` | `pan-client` | Identifies the registered application. The server looks up its allowed redirect URIs, scopes, and secret from this ID. |
| `redirect_uri` | `http://localhost:3000/callback` | Where the server sends the browser after login. Must exactly match a URI pre-registered for this client — prevents codes from being sent to attacker-controlled URLs. |
| `scope` | `openid profile email pan.read` | Permissions being requested. `openid` triggers OIDC and issues an ID token; `profile` / `email` add standard identity claims; `pan.read` is a custom application scope granting read access to Pan resources. |
| `state` | `random-csrf-token` | An opaque value generated by the client, echoed back unchanged in the redirect. The client verifies it matches before accepting the response — prevents CSRF attacks. Must be a cryptographically random value in production. |
| `code_challenge_method` | `S256` | Hashing algorithm used to derive the challenge from the verifier. `S256` (SHA-256) is required; plain-text is not accepted. |
| `code_challenge` | derived value | SHA-256 hash of the `code_verifier`, base64url-encoded. Sent in the authorization request; verified server-side during token exchange. |

The user is shown a login page (`bob@codingrodent.com` / `password1` with the default in-memory store).  
After login and consent, they are redirected back to your app:

```
http://localhost:3000/callback?code=AUTH_CODE&state=random-csrf-token
```

### Step 3 — Exchange the code for tokens

Include the `code_verifier` from Step 1. The server hashes it and compares it against the `code_challenge` sent in Step 2 — if they match, the token is issued.

```bash
curl -X POST http://localhost:9095/oauth2/token \
  -u pan-client:secret \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=authorization_code" \
  -d "code=AUTH_CODE" \
  -d "redirect_uri=http://localhost:3000/callback" \
  -d "code_verifier=$CODE_VERIFIER"
```

Response:

```json
{
  "access_token": "eyJ...",
  "refresh_token": "eyJ...",
  "id_token": "eyJ...",
  "token_type": "Bearer",
  "expires_in": 300,
  "scope": "openid profile email pan.read"
}
```

### Step 4 — Call a protected endpoint

```bash
curl http://localhost:9095/user/v1/me \
  -H "Authorization: Bearer ACCESS_TOKEN"
```

### Step 5 — Refresh the access token

```bash
curl -X POST http://localhost:9095/oauth2/token \
  -u pan-client:secret \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=refresh_token" \
  -d "refresh_token=REFRESH_TOKEN"
```

## Grant type 2 — Client Credentials (machine-to-machine)

Use this when there is no user — a backend service calling another backend service.

```bash
curl -X POST http://localhost:9095/oauth2/token \
  -u pan-client:secret \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=client_credentials" \
  -d "scope=pan.read"
```

Response:

```json
{
  "access_token": "eyJ...",
  "token_type": "Bearer",
  "expires_in": 300,
  "scope": "pan.read"
}
```

Use the `access_token` in the `Authorization: Bearer` header on every request.

## Grant type 3 — Username/Password login (`/login` endpoint)

Use this when a client wants to exchange a username and password directly for a JWT without going through the OAuth2 authorization code flow. This is a non-standard, application-specific endpoint — suitable for trusted first-party clients (e.g. a native mobile app or a CLI tool) where redirecting through a browser is not practical.

The token is signed with the same RS256 key pair as the OAuth2 tokens, so it can be verified by any resource server using the `/oauth2/jwks` endpoint.

### Request

```
POST http://localhost:9095/user/v1/login
Content-Type: application/json
x-correlation-id: <uuid>
```

```json
{
  "username": "bob@codingrodent.com",
  "password": "password1",
  "client_id": "pan-client"
}
```

| Field | Required | Description |
|---|---|---|
| `username` | yes | The user's email address. |
| `password` | yes | The user's plaintext password. |
| `client_id` | yes | The client application identifier. Embedded in the token's `aud` claim. |

The `x-correlation-id` header is required on all requests and is used for request tracing in logs.

### curl example

```bash
curl -X POST http://localhost:9095/user/v1/login \
  -H "Content-Type: application/json" \
  -H "x-correlation-id: $(uuidgen)" \
  -d '{"username":"bob@codingrodent.com","password":"password1","client_id":"pan-client"}'
```

### Response

On success the endpoint returns `200 OK` with the JWT as a plain string in the response body (not JSON-wrapped):

```
eyJhbGciOiJSUzI1NiJ9.eyJqdGkiOiI...
```

On failure (wrong username or password) the endpoint returns `401 Unauthorized` with an empty body.

### JWT payload claims

| Claim | Description |
|---|---|
| `jti` | Unique token ID (UUID). |
| `iss` | Issuer — `http://localhost:9095`. |
| `sub` | Subject — the authenticated user's email address. |
| `aud` | Audience — the `client_id` supplied in the request. |
| `scope` | Array of role names granted to the user (e.g. `["ADMIN","INTERACTIVE"]`). |
| `iat` | Issued at — Unix timestamp when the token was created. |
| `exp` | Expiration — Unix timestamp 1 hour after `iat`. |

## Token structure

All tokens are **JWTs** signed with **RS256** (RSA + SHA-256). The `kid` in the JWT header identifies which public key from `/oauth2/jwks` was used to sign the token, enabling key rotation.

### Token endpoint response fields

| Field | Present in | Description |
|---|---|---|
| `access_token` | All grants | JWT used to call protected API endpoints. Send as `Authorization: Bearer <token>`. Expires in 300 s. |
| `refresh_token` | `authorization_code` only | Opaque token used to obtain a new `access_token` without re-authentication. |
| `id_token` | `authorization_code` with `openid` scope | JWT containing identity claims about the authenticated user. For the client to consume, not for API calls. |
| `token_type` | All grants | Always `Bearer`. |
| `expires_in` | All grants | Lifetime of the `access_token` in seconds (300). |
| `scope` | All grants | Space-separated list of scopes actually granted. |

### Access token JWT payload

| Claim | Description |
|---|---|
| `sub` | Subject — the authenticated user's identifier (e.g. `alice`). |
| `aud` | Audience — the `client_id` this token was issued for (`pan-client`). |
| `iss` | Issuer — the authorization server that issued the token (`http://localhost:9095`). |
| `iat` | Issued at — Unix timestamp when the token was created. |
| `nbf` | Not before — earliest time the token is valid (equal to `iat`; valid immediately). |
| `exp` | Expiration — Unix timestamp after which the token must be rejected. |
| `jti` | JWT ID — unique identifier for this token; used to detect replay attacks. |
| `scope` | Array of granted scopes (e.g. `["pan.read", "openid", "profile", "email"]`). |

### ID token JWT payload

The ID token is an OIDC-specific token consumed by the client to learn who the user is. It shares several standard JWT claims with the access token and adds OIDC-specific ones:

| Claim | Description |
|---|---|
| `sub` | Subject — the authenticated user's identifier. |
| `aud` | Audience — the `client_id` this token was issued for. |
| `azp` | Authorized party — the client that requested the token (same as `aud` in single-client setups). |
| `iss` | Issuer — the authorization server that issued the token. |
| `iat` | Issued at — Unix timestamp when the token was created. |
| `exp` | Expiration — Unix timestamp after which the token must be rejected (1800 s lifetime). |
| `jti` | JWT ID — unique identifier for this token. |
| `auth_time` | Unix timestamp of when the user actually authenticated (may predate `iat` if using an existing session). |
| `sid` | Session ID — identifies the user's login session; used for logout / session management. |

## Verifying tokens (resource server setup)

Any other Spring Boot service that needs to accept tokens issued by this server should add:

```gradle
implementation "org.springframework.boot:spring-boot-starter-oauth2-resource-server:${springBootVersion}"
```

And in its `application.properties`:

```properties
spring.security.oauth2.resourceserver.jwt.issuer-uri=http://localhost:9095
```

Spring will fetch the public keys from `/oauth2/jwks` automatically and validate incoming JWTs without any additional code.

## Java client example (Spring's RestClient)

```java
// Step 1: obtain a token (client credentials shown here)
RestClient authClient = RestClient.create();

MultiValueMap<String, String> form = new LinkedMultiValueMap<>();
form.add("grant_type", "client_credentials");
form.add("scope", "pan.read");

Map<String, Object> tokenResponse = authClient.post()
        .uri("http://localhost:9095/oauth2/token")
        .headers(h -> h.setBasicAuth("pan-client", "secret"))
        .contentType(MediaType.APPLICATION_FORM_URLENCODED)
        .body(form)
        .retrieve()
        .body(new ParameterizedTypeReference<>() {});

String accessToken = (String) tokenResponse.get("access_token");

// Step 2: call a protected endpoint
RestClient apiClient = RestClient.create();

Map<String, Object> me = apiClient.get()
        .uri("http://localhost:9095/user/v1/me")
        .header("Authorization", "Bearer " + accessToken)
        .retrieve()
        .body(new ParameterizedTypeReference<>() {});
```

## Python client example (requests)

```python
import requests

# Step 1: obtain a token (client credentials)
token_response = requests.post(
    "http://localhost:9095/oauth2/token",
    auth=("pan-client", "secret"),
    data={
        "grant_type": "client_credentials",
        "scope": "pan.read",
    },
)
access_token = token_response.json()["access_token"]

# Step 2: call a protected endpoint
response = requests.get(
    "http://localhost:9095/user/v1/me",
    headers={"Authorization": f"Bearer {access_token}"},
)
print(response.json())
```

## Generating a new key pair

The authorization server loads its signing keys from `src/main/resources/app.key` (private) and `src/main/resources/app.pub` (public). Run these commands from the `src/main/resources/` directory to replace them:

```bash
# Generate a 2048-bit RSA private key and convert to PKCS#8 (required by Spring's RsaKeyConverters)
openssl genrsa -out app.key.pkcs1 2048
openssl pkcs8 -topk8 -inform PEM -outform PEM -nocrypt -in app.key.pkcs1 -out app.key
rm app.key.pkcs1

# Extract the public key
openssl rsa -in app.key -pubout -out app.pub
```

> **Warning:** Replacing the key pair invalidates all previously issued tokens. Users and services holding existing JWTs will receive 401 errors until they obtain new tokens.

The `kid` (key ID) embedded in issued JWTs is derived from the key thumbprint, so it changes automatically when the key pair is replaced — no manual configuration needed.

`app.key` contains the private key and must never be committed to version control. In production, load it from a secret store (e.g. Vault, an environment-injected file, or a Kubernetes secret) rather than bundling it in the JAR.
